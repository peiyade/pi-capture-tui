# AI-Powered Vibe Coding 记录

## 项目背景

这是一个使用 Claude Code（Claude Sonnet 4.6）进行"Vibe Coding"的实战记录。

**Vibe Coding**: 让 AI 处理实现细节，人类专注于产品愿景和交互设计。

## 项目目标

创建一个 TUI（Terminal User Interface）工具，用于快速捕获想法并保存到 Obsidian，同时配备一个温和的 AI 秘书进行简短的共鸣式反馈。

## 开发时间线

### Session 1: 基础架构（约 2 小时）

**人类输入**：
> "我想做一个类似 pi-capture-obsidian 的 TUI 工具，有三个框：历史框、秘书框、输入框。支持中文输入和 Emacs 键位。"

**AI 实现**：
- 搭建 Rust + ratatui 项目结构
- 实现三框布局（历史/秘书/输入）
- 添加基础键盘事件处理
- 实现 Obsidian Dataview 格式的 Markdown 输出

**关键决策**：
- 使用 `ratatui` 而非 `tui-rs`（前者更活跃）
- 历史记录按时间正序排列（旧在上，新在下）
- 支持多行输入（Shift+Enter 换行）

### Session 2: AI 秘书功能（约 1.5 小时）

**人类输入**：
> "秘书应该在用户提交想法后才给出反馈，而不是输入过程中。不要关键词判断，用 LLM 生成共鸣式感叹。"

**AI 实现**：
- 重构 AI 触发逻辑（从 on_input_changed 改为 submit_entry）
- 接入 Stepfun API（OpenAI-compatible）
- 添加可配置 personality（soul/prompt）
- 右下角显示 `——<秘书名>` 格式

**遇到的坑**：
- macOS config 路径是 `~/Library/Application Support/` 而非 `~/.config/`
- API key 从 `.env` 加载但环境变量未设置，最终改为配置文件硬编码（本地使用）

### Session 3: 交互优化（约 1 小时）

**人类输入**：
> "增加 :h 帮助页面，简化状态栏，显示模型名和连通性。"

**AI 实现**：
- 添加 Vim-like 命令模式（:q / :h）
- 简化状态栏为 `:q | :h | model | status`
- 修复所有 compiler warnings
- 更新 README 文档

## 有效 Prompt 模式

### 1. 约束优先
❌ "添加 AI 功能"
✅ "AI 在提交后触发，不做关键词判断，20字以内共鸣式感叹"

### 2. 具体示例
❌ "支持快捷键"
✅ ":q 退出, :h 帮助, Emacs 键位 Ctrl+N/P/F/B"

### 3. 上下文保持
- 主动提及之前的状态（"之前我们决定..."）
- 引用具体文件/行号

### 4. 及时修正
当 AI 误解时，立即用 "不是...而是..." 句式纠正

## 反思

### 什么工作得好
1. **快速原型**: AI 能在几分钟内生成可运行的代码框架
2. **细节处理**: UTF-8 边界、异步处理、错误处理等容易遗漏的点
3. **多语言支持**: 中文显示宽度计算等

### 什么需要改进
1. **上下文窗口**: 长会话后需要提醒 AI 之前的决策
2. **测试**: AI 不擅长设计测试用例，需要人工补充
3. **性能优化**: AI 倾向于生成"能工作"而非"最优"的代码

## 结论

Vibe Coding 适合：
- ✅ 个人工具/原型开发
- ✅ 明确需求的小项目
- ✅ 探索性实现（"试试这个功能"）

Vibe Coding 不适合：
- ❌ 需要严格安全审计的代码
- ❌ 复杂状态机的逻辑
- ❌ 性能关键路径（需要人工 profiling）
